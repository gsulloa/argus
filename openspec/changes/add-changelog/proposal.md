## Why

Argus versions itself on every release (`VERSION` bump across `tauri.conf.json` / `package.json` / `Cargo.toml`, tag `vX.Y.Z`, GitHub Release), but there is no **human-readable, curated** record of what changed between versions. The only changelog today is `packages/app/CHANGELOG.md`, auto-generated by `auto-changelog` from raw commit/PR titles — it is noisy, not organized by change type, buried inside a package folder, and never surfaced to users inside the app. Users who auto-update have no way to learn what's new; the team has no single, editorial source of truth for release notes.

This change introduces one curated changelog as the single source of truth, consumed identically by GitHub Releases and by an in-app viewer.

## What Changes

- **Create a curated `CHANGELOG.md` at the repo root** following [Keep a Changelog](https://keepachangelog.com/) + [SemVer](https://semver.org/): an `[Unreleased]` section on top, then one `## [X.Y.Z] - YYYY-MM-DD` section per version, each grouped into `Added` / `Changed` / `Fixed` / `Removed`. This becomes the **single source of truth** for release notes.
- **Backfill** the recent version history (at least the `v0.7.x` line, ideally back to `v0.1.x`) into that file from existing tags / PRs / the current auto-generated changelog.
- **Formalize changelog maintenance in the release flow**: the version bump step (`scripts/bump-version.mjs` / `scripts/release.sh`) converts the `[Unreleased]` section into a dated `## [X.Y.Z]` section and re-opens an empty `[Unreleased]`. Contributors edit `[Unreleased]` in their PRs.
- **Retire the auto-generated changelog**: remove (or freeze as an archived historical file) `packages/app/CHANGELOG.md` and the `changelog` npm script; **BREAKING** for the release workflow — `release.yml` release-note extraction is repointed at the root file and its new `## [X.Y.Z]` header format.
- **Add an in-app changelog viewer**: a command-palette entry ("Help: Show changelog" / "What's new") that opens a DESIGN.md-compliant view rendering the changelog bundled with the running build, current version highlighted, full history browsable.
- **Add a "What's new" prompt on update**: after auto-update installs a newer version, the viewer opens automatically on next launch scrolled to the changes since the previously seen version (dismissible, remembered via settings). No auto-generation from PR labels — entries are written by hand.

## Capabilities

### New Capabilities
- `changelog-maintenance`: the curated root `CHANGELOG.md` as single source of truth — its format contract (Keep a Changelog + SemVer), where it lives, how `[Unreleased]` is edited in PRs, and how the release/bump flow promotes `[Unreleased]` into a dated version section.
- `changelog-viewer`: the in-app experience — command-palette entry, rendering the bundled changelog per DESIGN.md, current-version highlight, full-history browsing, and the "What's new since last update" auto-open behavior gated on a last-seen-version setting.

### Modified Capabilities
- `github-release-publishing`: the requirement "Release notes come from the changelog with an auto-generated fallback" changes its source of truth from `packages/app/CHANGELOG.md` (auto-changelog `#### [vX.Y.Z]` headers) to the root `CHANGELOG.md` (Keep a Changelog `## [X.Y.Z] - date` headers); the extraction logic in `release.yml` is updated accordingly.

## Impact

- **New file**: `/CHANGELOG.md` (repo root).
- **Removed / archived**: `packages/app/CHANGELOG.md`, `changelog` script in `packages/app/package.json`, and the `auto-changelog` dev dependency.
- **Release tooling**: `packages/app/scripts/release.sh` and/or `bump-version.mjs` (promote `[Unreleased]`), `.github/workflows/release.yml` (release-note extraction awk repointed to root + new header regex).
- **Frontend**: new changelog viewer under `packages/app/src/platform/` (component + command registration), bundled copy of the changelog synced at build time (new prebuild sync step + Vite `?raw` import), last-seen-version persisted via the existing `platform/settings` localStorage pattern, current version read via `@tauri-apps/api/app` `getVersion()` (already used by `UpdaterProvider`).
- **Build**: `dev` / `build` npm scripts gain a changelog-sync prebuild step; the synced copy is gitignored.
- **Docs**: `README.md` release section and `/ship` skill notes reference the new single-source changelog flow.
- **No runtime backend (Rust) changes**; no new Tauri command required (changelog is bundled at build time, not read from disk at runtime).
