## Why

Argus ya cubre el flujo de browse → view → edit sobre tablas, pero todo edit directo en grid es un subset del trabajo real con Postgres. Cualquier query no trivial (joins, agregaciones, DDL, EXPLAIN, scripts ad hoc) hoy obliga a saltar a `psql` o a TablePlus. Esta rebanada cierra la brecha: una pestaña dedicada con editor SQL real (CodeMirror 6 + dialecto Postgres), atajos para ejecutar la statement bajo el cursor, autocomplete sobre el schema ya cargado en sidebar, y un viewer de resultados que reusa el grid existente. Con esto Argus deja de ser "TablePlus de tablas" y pasa a ser una herramienta SQL de uso diario.

## What Changes

- Nueva pestaña kind `postgres-query` con un editor CodeMirror 6 por instancia. Cada pestaña mantiene su propio buffer de SQL, su selección, su resultado más reciente, y vive aislada del resto.
- Editor con highlight Postgres (`@codemirror/lang-sql` con `PostgreSQL` dialect), tema integrado a `DESIGN.md` (fuente `Geist Mono`, paleta sobria, gutter sutil), keymap por defecto + atajos custom listados abajo.
- Atajo `⌘↩` ejecuta el SQL: si hay selección, ejecuta la selección; si no, ejecuta la statement bajo el cursor. La detección de "statement bajo el cursor" SHALL hacerse splitting por `;` respetando strings y comentarios `--` / `/* */` para no cortar literales.
- Multi-statement: cuando el SQL ejecutado contiene más de una statement separada por `;`, las ejecuta en orden secuencial sobre la **misma conexión** (no transacción implícita — cada una commitea por sí sola, igual que `psql`); el panel de resultado renderiza una sub-pestaña por cada resultado en orden.
- Comando IPC `postgres_run_sql(connectionId, sql, origin?)` que ejecuta una sola statement y retorna `{ kind: "rows", columns, rows, query_ms, truncated_columns } | { kind: "affected", command_tag, affected_rows, query_ms }`. `kind: "rows"` para `SELECT` / `RETURNING`; `kind: "affected"` para todo lo demás (`INSERT`/`UPDATE`/`DELETE`/`DDL`/`SET`/etc.).
- Comando IPC `postgres_run_sql_many(connectionId, statements, origin?)` que recibe un array de SQL parseado por el frontend y devuelve `Array<{ statement_index, status: "ok"|"err", result?: <RunSqlResult>, error? }>`. Se ejecuta secuencialmente; si una falla, las posteriores NO se ejecutan, pero las previas que ya commitearon NO se revierten (ese es el contrato de Postgres con statements separadas).
- Read-only enforcement: si la conexión es `read_only: true`, ambos comandos pasan por la heurística existente del pool (`is_mutating_sql`) — los `SELECT` y `EXPLAIN` puros pasan; cualquier statement detectada como mutación es rechazada con `AppError::Validation "connection is read-only"`. La UI surface eso con un banner inline encima del editor.
- Panel de resultado debajo del editor:
  - Para `kind: "rows"`: reusa la grid virtualizada existente del módulo `postgres-data-grid` en **modo read-only adhoc** (sin sort/filter server-side, sin paginación scroll-to-load — los resultados de un query libre vienen completos en una sola respuesta hasta un cap). El grid muestra columnas, valores, y honra el inspector de fila (panel del shell) cuando se selecciona una fila.
  - Para `kind: "affected"`: un mini-summary "X rows affected · 12 ms · `INSERT 0 3`".
  - Para errores: bloque danger con mensaje + SQLSTATE + posición (cuando Postgres la entrega) + botón "Show in editor" que mueve el cursor a esa posición.
- Cap de filas devueltas por `postgres_run_sql`: 10 000 filas por defecto; al llegar al cap, la respuesta marca `truncated: true` y la UI muestra un banner "Result truncated at 10,000 rows — add LIMIT or scroll-to-load is not supported here". El cap es configurable (no en V1) — por ahora hardcoded.
- Autocomplete: el editor consulta el cache de `postgres-schema-browser` ya cargado en memoria para sugerir nombres de schemas, tablas/views/matviews y columnas (cuando se ha cargado `postgres_list_table_extras` o se conoce la columna por una query previa). El autocomplete NO dispara fetches nuevos a Postgres; opera sobre lo que el sidebar ya cargó. Si el schema activo no se ha expandido, el autocomplete se degrada silenciosamente y solo sugiere keywords SQL.
- Indicador en la barra inferior del editor: filas devueltas / `X rows affected`, `query_ms`, indicador de truncation cuando aplica. Los detalles completos (SQL ejecutado, params, code) viven en el `activity-log`.
- Persistencia ligera del buffer del editor: cada pestaña SHALL persistir su SQL bajo settings key `pgQueryBuffer:<tabId>` (debounced 500ms). Al reabrir la app, las pestañas `postgres-query` se restauran con su SQL — ese contrato lo respeta el `app-shell` tab system existente para tab payloads, sin cambios; solo definimos que el payload incluye `sql: string`. **Nota**: el shell hoy NO persiste tabs entre sesiones. Esa persistencia queda OUT OF SCOPE de este change; el buffer SOLO sobrevive dentro de la sesión (cambios de pestaña y refrescos del editor) — no entre relaunches.
- Múltiples pestañas de query simultáneas: cada pestaña tiene un id estable autogenerado `pgquery:<connectionId>:<uuid>`. Diferentes pestañas pueden apuntar al mismo connection sin conflicto (cada una toma su client del pool al ejecutar).
- Comandos en command-palette:
  - `SQL: New Query` (general) → si hay una conexión Postgres focuseada en sidebar, abre una pestaña nueva contra esa conexión; si no, transiciona a chooser de conexiones activas.
  - `SQL: New Query Here` (contextual desde la fila de conexión / desde un schema / desde una tabla) → abre una pestaña con un placeholder del SQL pre-poblado: para conexión vacío, para schema `SET search_path TO "<schema>";\n\n`, para tabla `SELECT * FROM "<schema>"."<table>" LIMIT 100;`.
- Entry point desde el sidebar: cada conexión activa SHALL exponer un botón `+ Query` (visible al hover sobre la fila de conexión, junto al refresh existente) que dispara `SQL: New Query` para esa conexión. Es el path principal — la palette es secundaria.
- Activity-log: nuevo `kind: "run_sql"` con `sql: <SQL ejecutado>`, `params: null` (no hay binds, este flujo no acepta binds desde UI en V1), `metric: { kind: "rows", value: <row count> }` para `kind: "rows"` o `{ kind: "affected", value: <affected_rows> }` para `kind: "affected"`. El origen es siempre `"user"` (todo run-sql nace de una activación humana).
- Welcome / placeholder cuando no se ha ejecutado nada: el panel de resultado muestra un hint "Press ⌘↩ to run". El editor arranca con foco al abrir la pestaña.

## Capabilities

### New Capabilities

- `postgres-sql-editor`: pestaña `postgres-query`, editor CodeMirror con dialecto Postgres + autocomplete schema-aware, runtime de ejecución (single y multi-statement con detección por cursor), panel de resultados (grid adhoc para `kind: "rows"`, summary para `kind: "affected"`, error block con SQLSTATE + posición), comandos `postgres_run_sql` y `postgres_run_sql_many`, integración con `command-palette` y entry point en sidebar, persistencia in-session del buffer, banner read-only.

### Modified Capabilities

- `postgres-data-grid`: la grid virtualizada SHALL exponer un sub-componente reusable `<AdhocResultGrid columns rows />` que la `postgres-sql-editor` consume para renderizar resultados ad hoc. Ese sub-componente NO incluye sort/filter server-side ni scroll-to-load (es para un dataset completo en memoria); reusa el render de celda, la virtualización vertical, el styling, y la integración con el inspector. La grid editable existente NO se ve afectada — sigue siendo la del flujo `postgres-table-data`.
- `activity-log`: el set de `kind` se amplía con `"run_sql"`. Mapping de metric añade `run_sql → { kind: "rows", value: <returned row count> }` para resultados con filas, `{ kind: "affected", value: <affected> }` para statements no-SELECT. Esto requiere un nuevo variant en el discriminated union de `metric` (`{ kind: "affected", value: number }`).
- `command-palette`: el módulo Postgres registra dos comandos nuevos (`SQL: New Query`, `SQL: New Query Here`); no hay cambio en el contrato de la palette en sí, pero el spec de `postgres-schema-browser` enumera comandos del módulo Postgres y debe extenderse para cubrir estos.
- `postgres-schema-browser`: añade el botón `+ Query` por conexión activa en el sidebar. Es un cambio menor del UI del sidebar — el comportamiento del schema tree no cambia. También añade los dos comandos enumerados arriba al spec de palette commands del schema browser (donde hoy viven `Schema: Refresh` y `Schema: Filter Visible…`).

## Impact

- **Backend**: nuevo módulo `src-tauri/src/modules/postgres/sql.rs` con `postgres_run_sql` y `postgres_run_sql_many`. Reusa `executeQuery` y `executeMutation` del pool según el shape detectado (`is_mutating_sql` ya existe del flujo edit). Reusa el envelope de `CellValue` (binary/truncated) del data viewer. Reusa la emisión de `argus:activity-log` con el nuevo kind. Cap de 10 000 filas y truncation marker.
- **Frontend**: nuevo subdirectorio `src/modules/postgres/sql/` con `QueryTab.tsx`, `QueryEditor.tsx` (envuelve CodeMirror), `useQueryAutocomplete.ts` (lee del cache del schema browser), `useQueryRun.ts` (estado del último run, multi-statement), `ResultPanel.tsx`, `ResultErrorBlock.tsx`, `MultiStatementTabs.tsx`, `useQueryBuffer.ts` (persistencia in-session). Modificar `src/modules/postgres/index.ts` para registrar los comandos de palette y el tab kind. Modificar `src/modules/postgres/schema/SchemaTree.tsx` para mostrar el botón `+ Query` en cada fila de conexión activa.
- **Reuso de grid**: extraer `<AdhocResultGrid />` desde el actual `DataGrid.tsx` (probablemente un split en dos componentes: el wrapper editable actual y un core read-only adhoc) para no acoplar el editor al ciclo de vida del table viewer.
- **Atajos**: `⌘↩` ejecuta (en pestaña `postgres-query` con el editor focuseado o el panel de resultado). `Escape` quita foco de un input modal pero no se usa para nada custom en esta pestaña; el editor mantiene su keymap por defecto. `⌘W` cierra la pestaña — si el buffer no está vacío y no se ha ejecutado nada, NO confirmamos (es solo SQL no commiteado a BD; menos crítico que un edit buffer dirty).
- **Settings**: ninguna nueva persistente; `pgQueryBuffer:<tabId>` se gestiona in-memory por la sesión (puede ir al `settings` storage para futuras sesiones, pero como el shell aún no restaura tabs entre relaunches, no aporta nada).
- **Dependencias nuevas**: `codemirror` (`@codemirror/state`, `@codemirror/view`, `@codemirror/commands`, `@codemirror/language`, `@codemirror/autocomplete`, `@codemirror/search`) + `@codemirror/lang-sql` con dialecto `PostgreSQL`. Sin uso de `react-codemirror` wrapper — montamos el editor directamente vía `useEffect` sobre un `<div ref>` (consistente con cómo el resto de la app evita wrappers innecesarios).
- **Out of scope**: EXPLAIN visualizer (un siguiente change), format SQL (mismo), snippets / templates de usuario, query parameters (binds vía `$1`/`$2` en UI), import/export de SQL, history (es change #7 en el roadmap), saved queries / favoritos. PartiQL / Dynamo es V2.
- **Riesgos**: el detector de "statement bajo el cursor" puede equivocarse en SQL con `;` dentro de strings; mitigación con un splitter consciente de `'…'`, `$$…$$`, comentarios. La detección de mutación heurística (`is_mutating_sql`) puede tener falsos positivos en strings que parezcan DML (ej. `SELECT 'UPDATE foo'`); mitigación: el splitter ya retira comentarios y respeta strings, y el helper se aplica sobre la statement ya sanitizada — y el peor caso es un falso positivo en read-only que el usuario sortea quitando read-only por sesión (no es un agujero de seguridad). El cap de 10k filas evita romper la UI con `SELECT * FROM huge_table`.
