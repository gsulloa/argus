## Context

`release.yml` runs on `push: tags: v*`. The `build` matrix produces four platform
bundles; the `publish` job downloads all four artifact sets, runs a "Verify all four
bundles present" gate, builds `latest.json` / `download.json`, and uploads everything
to S3 + CloudFront. The `publish` job currently declares `permissions: { id-token:
write, contents: read }`. There is no GitHub Release created anywhere, and
`packages/app/CHANGELOG.md` already keeps per-version sections (header form
`#### [vX.Y.Z](...compare...)`). Issue #113 asks for a tag-driven GitHub Release with
notes and the installers attached.

## Goals / Non-Goals

**Goals:**
- Create/Update a GitHub Release for each `vX.Y.Z` tag, gated on the existing
  bundle-verification success.
- Populate notes from the matching `CHANGELOG.md` section, with a deterministic
  auto-generated fallback.
- Attach the four end-user installers (`.dmg` ×2, `.AppImage`, `.msi`) as assets.
- Be idempotent on workflow re-runs for the same tag.

**Non-Goals:**
- Changing the S3 + CloudFront publishing path or the manifest schemas.
- Authenticode / extra code-signing (tracked elsewhere).
- Rewriting the changelog generation (`auto-changelog`) — we only *read* it.
- Attaching updater archives or `.sig` files as Release downloads.

## Decisions

### D1 — Add the Release step inside the existing `publish` job (not a new job)
The installers are already staged in `publish`'s `staging/` directory after the
verify step, and the job already resolves `version` from the tag. Adding the step
there avoids re-downloading artifacts into a second job. The step is placed **after**
"Verify all four bundles present" so the gate naturally protects it.
*Alternative considered:* a sibling job `release` gated on `needs: [build]` — rejected
because it would have to re-download the four artifacts just to attach them.

### D2 — Use `softprops/action-gh-release` (pinned) over raw `gh release create`
`softprops/action-gh-release@v2` is idempotent by design (creates or updates the
Release for the tag), takes a glob of `files:` to attach, and accepts a `body` /
`body_path`. This directly satisfies the idempotency and asset-attach requirements
with less shell than scripting `gh release create || gh release edit` + per-file
`gh release upload --clobber`.
*Alternative considered:* `gh release create vX.Y.Z ... || gh release edit` — more
moving parts and easy to get the create-vs-edit branching wrong; rejected.

### D3 — Notes: extract the CHANGELOG section, fall back to autogenerated
A small bash/node step reads `packages/app/CHANGELOG.md` and slices the block for the
current `vX.Y.Z` header up to the next `####` header, writing it to `RELEASE_NOTES.md`.
If no section matches (empty slice), set the action's `generate_release_notes: true`
so GitHub auto-generates notes from commits since the previous tag. This keeps the
notes deterministic for a given tag + changelog state.
*Alternative considered:* always autogenerate — rejected because the curated
changelog is more meaningful than a raw commit list when it exists.

### D4 — Grant `contents: write` to the `publish` job
Creating a Release via `GITHUB_TOKEN` requires `contents: write`. The job keeps
`id-token: write` (needed for AWS OIDC). Scope stays at job level, not workflow level.

### D5 — Attach only the four installers
The `files:` glob targets `staging/Argus_*_*.dmg`, `staging/Argus_*_x64.AppImage`,
`staging/Argus_*_x64.msi` explicitly — never `*.sig` / `*.tar.gz` / `*.msi.zip` — so
the Release page shows clean end-user downloads matching `download.json`.

## Risks / Trade-offs

- **CHANGELOG section parsing is brittle** (header format could drift) → fallback to
  `generate_release_notes` guarantees a non-empty body even if extraction returns
  nothing; the slice is anchored on the literal `vX.Y.Z` string from the tag.
- **Double source of truth** (Release assets vs S3 installers could diverge) → both
  are produced from the *same* `staging/` files in the same job run, so they cannot
  diverge for a given tag.
- **Permission widening** (`contents: write`) → scoped to the `publish` job only;
  the token still cannot push code, only manage Releases/assets for this repo.
- **Re-run after a partial manual fix** → `action-gh-release` updates in place;
  re-running with the full `staging/` set re-uploads/overwrites the four assets.

## Migration Plan

1. Add `contents: write` to the `publish` job permissions.
2. Add a notes-extraction step writing `RELEASE_NOTES.md` (with empty-detection).
3. Add the `softprops/action-gh-release@v2` step after the verify gate, attaching the
   four installers and using `body_path` or `generate_release_notes`.
4. Validate by pushing a throwaway pre-release tag (e.g. `v0.0.0-relnotes1`) and
   confirming the Release appears with notes + four assets; delete the test Release
   and tag afterward.

Rollback: removing the step leaves the rest of `publish` (S3/CloudFront) untouched —
the Release feature is additive and isolated.
