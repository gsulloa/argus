## Context

Each SQL editor runs a query by `invoke`-ing an engine-specific Tauri command and showing a `running` state until it resolves. There is no way to abort a run in flight.

The backend cancel primitives already exist per engine, but their wiring differs:

- **Postgres** (`modules/postgres/sql.rs`) — `postgres_run_sql` obtains `client.cancel_token()` and calls `fire_cancel(...)`, but **only inside the 60s `RUN_SQL_TIMEOUT` branch**. The code comment literally notes "A real cancel button is follow-up."
- **MySQL** (`modules/mysql/sql.rs` + `mysql/cancel.rs`) — `capture_thread_id()` and `fire_mysql_cancel()` (`KILL QUERY <thread_id>`) exist, but the captured `thread_id` is discarded (`let _ = thread_id; // captured for potential cancel integration`).
- **MSSQL** (`modules/mssql/sql.rs` + `mssql/cancel.rs`) — `run_cancellable_query()` captures `@@SPID` and has `fire_mssql_cancel()` (KILL on a fresh connection), used only on timeout. `ActiveMssqlPool` already caches params + password for that path.
- **Athena** (`modules/athena/sql.rs`) — `athena_cancel_query` command + `athena:query-started` event already exist and `athenaApi.cancelQuery` is defined; but `useAthenaQueryRun` does not expose `cancel()` and the toolbar renders no Stop button.
- **CloudWatch** (`modules/cloudwatch/insights/`) — **complete**: `cloudwatch_cancel_insights` + `cloudwatch:query-started` event + `useQueryRun.cancel()` + the Run/Cancel toolbar button. This is the reference implementation.
- **DynamoDB** — scan/query/PartiQL has no server-side abort API.

Frontend state lives in per-engine `useQueryRun` hooks with a `RunState = idle | running | (done & …)` union and a `run({connectionId, …})` method; toolbars share `QueryTab` structure and DESIGN-compliant `toolbarButton` styling.

## Goals / Non-Goals

**Goals:**
- A visible Stop affordance (button + keyboard shortcut) appears while a query runs in the Postgres, MySQL, MSSQL, and Athena SQL editors; CloudWatch already has it.
- Triggering Stop aborts the in-flight statement in the backend and returns the editor to idle with a neutral "Query cancelled" outcome — no error block, no half-rendered rows, no hang.
- Reuse each engine's existing cancel primitive; add no new dependencies.
- Be robust to the obvious races (cancel after completion, double-cancel, cleanup on error/panic).

**Non-Goals:**
- DynamoDB cancellation (no server-side abort exists) — documented as a known limitation; its editors keep today's behavior.
- Cancelling background/non-user runs (autocomplete probes, schema introspection, history writes) — only user-initiated editor runs are cancellable.
- Changing query timeout values or the multi-statement execution semantics beyond "stop the batch when cancelled."

## Decisions

### 1. Correlate a run with a frontend-generated run token (sync engines)
For Postgres/MySQL/MSSQL the run command gains a `run_token: Uuid` argument generated by the frontend before `invoke`. The backend registers the cancel handle under that token; cancel calls `cancel_running_query(run_token)`.

- **Why over a backend `*:query-started` event** (the Athena/CloudWatch pattern): the synchronous engines have no server-assigned id, and a frontend-generated token is known *before* the run resolves, so there is no event round-trip and no race where the user clicks Stop before the started-event arrives. Athena/CloudWatch keep their event-based path because their cancellable id (the AWS execution/query id) only exists after `StartQuery`.

### 2. One shared in-flight registry + one `cancel_running_query` command
Introduce a shared `RunningQueryRegistry` (Tauri managed state): `Mutex<HashMap<Uuid, CancelEntry>>`, where `CancelEntry` holds a type-erased async cancel thunk (`Box<dyn Fn() -> BoxFuture<'static, ()> + Send + Sync>`) plus a `cancel_requested: bool` flag. Each engine constructs the appropriate thunk (Postgres: fire the `CancelToken`; MySQL: `fire_mysql_cancel(params, secret, thread_id)`; MSSQL: `fire_mssql_cancel(spid, params, password)`). A single `cancel_running_query(run_token)` command looks up the entry, sets the flag, and awaits the thunk best-effort.

- **Why over per-engine registries + `*_cancel_sql` commands**: the handle types differ, but the *cancel action* is uniform once type-erased, so one registry and one command remove most of the per-engine plumbing. Each engine still owns thunk construction, so engine-specific concerns (fresh-connection cancel, password source) stay local.
- **Alternative considered**: storing the live `JoinHandle`/aborting the Tokio task. Rejected — aborting the task drops the driver connection without telling the server to stop the query (it keeps running server-side until done), which is exactly the "looks cancelled but isn't" bug. The driver cancel primitives stop the work server-side.

### 3. Cancellation is classified by intent, not by driver error code
When the run future errors, the run command reports a **cancelled** outcome iff `cancel_requested` was set for its token; otherwise it reports the error normally. We do not parse SQLSTATE `57014` (PG), error `1317` (MySQL), or the MSSQL KILL error.

- **Why**: intent-based classification is uniform across engines and immune to driver-specific or localized error text. The flag is authoritative.

### 4. Backend signals cancellation as a distinct, non-error result
The run command resolves with a recognizable cancelled signal (an `AppError` with code `"Cancelled"`, or equivalently a cancelled result variant). The frontend maps it to a neutral terminal `cancelled` state — **not** the red error block.

- **Why an error-code over a new result enum variant**: avoids threading a new variant through every `RunSqlResult`/multi-outcome type and matches how the hooks already catch `AppError`. The hook special-cases the `Cancelled` code into the neutral state.
- Multi-statement runs: on cancel, the currently-executing statement reports `cancelled` and remaining statements report `skipped`; the batch loop stops (checks `cancel_requested` between statements).

### 5. Lifecycle & cleanup via an RAII guard
The run command registers the token at entry and removes it via a drop guard, so the entry is cleaned up even if the query errors or panics. Registration happens synchronously before awaiting the query body; for the (unlikely) cancel-before-register case the cancel command leaves a tombstone the run checks before executing.

### 6. Frontend: extend the existing per-engine hook + mirror the CloudWatch toolbar control
Each `useQueryRun` gains `cancel()` and a terminal `cancelled` state. The toolbar Run button becomes a Run⇄Stop control (Stop shown while `status === "running"`), styled by mirroring the already-shipped, DESIGN-compliant CloudWatch `insights/Toolbar.tsx` Run/Cancel button. The existing run shortcut (`Mod-Enter`) gains a companion cancel shortcut (e.g. `Mod-.` / `Escape` while running) registered in the editor keymap.

## Risks / Trade-offs

- **[Fresh-connection cancel needs credentials at cancel time]** MySQL/MSSQL cancel opens a new connection requiring the password. → Capture params + secret into the `CancelEntry` thunk at run start (MSSQL pool already caches password for this; MySQL captures from the same source the run used), so cancel needs no keychain round-trip.
- **[Best-effort cancel may not land instantly]** Server-side abort can take a moment (or be ignored by a stuck backend), and Azure SQL proxy routing can swallow a KILL. → Mitigations already in the codebase (TCP-RST-on-drop + KILL fallback for MSSQL); the UI returns to idle on the cancelled signal regardless, and cancel is best-effort/idempotent so a no-op never wedges the UI.
- **[Race: cancel before register]** Token unknown when cancel fires. → Tombstone entry checked by the run before executing; worst case the run proceeds and the user clicks Stop again.
- **[Token leak]** A crashed run could leave a stale registry entry. → RAII drop guard removes the entry on every exit path; entries are tiny and keyed per run.
- **[Cancelled-vs-real-error misclassification]** If a query genuinely errors at the same instant the user cancels, intent-flag wins and it shows as cancelled. → Acceptable: the user asked to stop; the precise error is moot.
- **[DynamoDB has no cancel]** Users may expect a Stop button everywhere. → Out of scope; do not render a Stop affordance for Dynamo and note the limitation.

## Migration Plan

Purely additive: new managed state, one new command, a new optional `run_token` arg on three run commands, additive frontend state. No persisted state, no wire-format break, no rollback complexity. If disabled, the editors behave exactly as today.

## Open Questions
