## Why

The auto-updater silently fails in production: it detects new versions and shows the pending badge, but two paths are broken — (1) on quit (⌘Q / window close) the new binary is not applied, so the next launch still runs the old version; (2) clicking **"Install update & restart"** does nothing visible — the app neither installs the update nor relaunches. Without on-disk logs visible to the user, we have no way to diagnose the failure on shipped builds — `console.debug` from the renderer is lost in release, and Rust `tracing` output is buried in `~/Library/Logs/Argus/argus.log` that non-technical users will never find.

## What Changes

- **Fix quit-time apply**: rework the renderer's `beforeunload` interception so the pending update actually installs before the process exits. The current implementation calls `event.preventDefault()` then `window.close()` from a `.finally`, which on macOS is a race the OS frequently wins — the webview gets torn down before `install()` finishes. Move the install to a Tauri `RunEvent::ExitRequested` hook in Rust where we can block exit deterministically until install completes.
- **Fix "Install update & restart"**: surface the failure cause. The current handler swallows errors into `console.debug` and resets local state, so the user sees "Installing…" briefly then nothing. Pipe install/relaunch errors through `tracing::error!` in Rust and through a user-visible toast / dialog in the renderer, and ensure the `relaunch()` call actually fires by routing both `install` and `relaunch` through a dedicated Rust command that logs each step.
- **Add structured updater logging**: emit `tracing::info!` events at every updater milestone (check started, version found, download started, download complete, install started, install complete, relaunch invoked, error at step X) from a new `updater` Rust module. Renderer-side `console.debug` calls move to a new `logUpdater(level, msg, fields?)` helper that invokes a `log_updater_event` Tauri command so renderer events also land in `argus.log`.
- **Add in-app logs viewer**: new menu item **"View update logs…"** in the `VersionIndicator` dropdown (visible always, not gated on pending update) that opens a modal showing the last ~200 lines of `argus.log` filtered to `target=updater` (or contains "updater"), with a "Reveal in Finder" / "Open log folder" button and a "Copy to clipboard" button. The modal reads the log via two new Tauri commands (`updater_logs_tail`, `updater_logs_reveal`).
- **Allow user-initiated relaunch via single Rust path**: replace the renderer-driven `pendingRef.current.install()` + JS `relaunch()` with a Rust command `updater_install_and_restart` that takes the staged update handle, logs each step, and relaunches via `app.restart()`. The renderer-held `Update` object stays for the existing quit-time path.

## Capabilities

### New Capabilities
- `updater-logs`: User-facing visibility into updater behavior. Adds structured logs from both Rust and renderer to `argus.log` under `target=updater`, and an in-app modal viewer to read recent updater events without leaving the app. Reveals the log folder in the OS file manager and copies log text to clipboard.

### Modified Capabilities
- `app-updater`: tightens existing requirements that are currently broken. The "Pending update applies on app quit" requirement gains a scenario that the swap MUST complete before process exit (not race the OS), implemented via a Rust exit-requested hook. The "Install-and-restart via dropdown menu" requirement (added in the prior `install-update-restart-button` change) gains scenarios that errors are logged, surfaced to the user, and that the entire install-and-restart sequence runs in Rust through a single command. The "Update errors do not crash or block the app" requirement is unchanged in spirit but extended to require that every updater error path emits a `tracing` event with sufficient detail for post-hoc debugging.

## Impact

- **Frontend**: `src/platform/updater/UpdaterProvider.tsx` (replace JS install/relaunch with `invoke("updater_install_and_restart")`, replace `beforeunload` install with `invoke("updater_apply_pending_on_quit")` registration at startup, add toast/dialog for surfaced errors, add `logUpdater()` helper), `src/platform/updater/index.ts` (re-export new logs hook), `src/platform/shell/VersionIndicator.tsx` + `.module.css` (new "View update logs…" item; new modal `UpdaterLogsDialog.tsx`).
- **Rust**: new module `src-tauri/src/platform/updater/` with `commands.rs` (`updater_install_and_restart`, `updater_apply_pending_on_quit`, `updater_logs_tail`, `updater_logs_reveal`, `log_updater_event`) and `mod.rs` (state holding the pending `Update` handle, log-tail reader). `src-tauri/src/lib.rs` wires the new commands, the `RunEvent::ExitRequested` hook, and ensures `init_tracing` always emits to file (currently file logging is `release`-only — extend to also log when `RUST_LOG` is set in debug for testing). `Cargo.toml` may add `tauri-plugin-opener` (or use the existing `dialog` plugin's `show_in_folder` equivalent) if not already present for "Reveal in Finder".
- **Capabilities**: `src-tauri/capabilities/default.json` — ensure `process:allow-restart` remains; no change needed for the `app.restart()` Rust API, but add `opener:default` (or equivalent) if we use it for reveal-in-folder. Otherwise reveal via `std::process::Command` in Rust without a plugin.
- **Logs file**: `argus.log` continues to live at `app_log_dir()` (`~/Library/Logs/Argus/argus.log` on macOS). No format change beyond adding `target=updater` discipline.
- **No DB / migration impact.** No breaking changes to user-facing connection or query flows.
- **Risk**: the new `RunEvent::ExitRequested` hook can hang quit indefinitely if `install()` deadlocks. Mitigation: 10-second timeout that logs `error` and exits anyway (user already chose to quit). Risk: log file may be large; tailing the last N lines reads only the end of the file via a reverse-line iterator, not the whole file.
