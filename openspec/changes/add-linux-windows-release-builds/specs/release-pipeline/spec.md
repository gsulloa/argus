## MODIFIED Requirements

### Requirement: CI workflow builds, signs, and publishes on every merge to master

The repository SHALL include a GitHub Actions workflow that triggers on `push` to `master`. The workflow MUST execute the following steps in order, failing fast on any error:

1. Bump the patch version across `tauri.conf.json`, `tauri.beta.conf.json`, `package.json`, and `src-tauri/Cargo.toml` to the next patch number.
2. Commit the bump with message starting with `chore: bump version to v` and `[skip ci]` to prevent recursion, then push and tag.
3. Build a matrix of FOUR targets in parallel:
   - `aarch64-apple-darwin` on `macos-latest`
   - `x86_64-apple-darwin` on `macos-13` (or `macos-latest` with explicit target)
   - `x86_64-unknown-linux-gnu` on `ubuntu-22.04`
   - `x86_64-pc-windows-msvc` on `windows-latest`
4. For each macOS target: code-sign the bundle with the `Developer ID Application` certificate loaded from `APPLE_CERTIFICATE` (base64) using `APPLE_CERTIFICATE_PASSWORD`, then submit it to Apple notarization via `notarytool` using `APPLE_ID`, `APPLE_PASSWORD`, and `APPLE_TEAM_ID`, and staple the ticket on success.
5. For the Linux target: install the WebKitGTK + libsoup + libayatana-appindicator system dependencies required by Tauri before invoking `tauri build`. No code-signing is performed.
6. For the Windows target: invoke `tauri build` with the stable MSVC toolchain. No Authenticode signing is performed in this iteration (deferred to a follow-up change).
7. Sign each updater archive (`.app.tar.gz` on mac, `.AppImage.tar.gz` on Linux, `.msi.zip` on Windows) with the Ed25519 updater private key loaded from `TAURI_UPDATER_PRIVATE_KEY` and `TAURI_UPDATER_KEY_PASSWORD`, producing `.sig` files.
8. Upload all artifacts (per-target installers + updater archives + `.sig` files) plus the generated `latest.json` to the R2 bucket using credentials from `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`.

The build matrix MUST use `fail-fast: false` so a transient failure on one OS does not cancel the other three; the publish step MUST require all four builds to succeed before uploading the manifest.

#### Scenario: Successful merge produces a new release across all four targets

- **WHEN** a PR merges to `master` and the workflow completes without errors
- **THEN** R2 contains `latest.json` pointing to the new version with `platforms` entries for `darwin-aarch64`, `darwin-x86_64`, `linux-x86_64`, and `windows-x86_64`, each with a valid `.sig` field; and R2 also contains the per-platform installers (`.dmg`, `.AppImage`, `.msi`) for new team members.

#### Scenario: Bump step is idempotent against its own commit

- **WHEN** the workflow's bump commit triggers another `push: master` event
- **THEN** the bump step detects that the head commit already starts with `chore: bump version to v` and exits early without bumping again or starting a build.

#### Scenario: Notarization transient failure retries

- **WHEN** `notarytool submit` fails with a transient network or Apple-server error
- **THEN** the step retries up to 3 times with backoff before failing the workflow.

#### Scenario: Linux build runs on a clean Ubuntu runner

- **WHEN** the `x86_64-unknown-linux-gnu` matrix entry runs on a fresh `ubuntu-22.04` runner
- **THEN** the workflow installs `libwebkit2gtk-4.1-dev`, `libsoup-3.0-dev`, `libayatana-appindicator3-dev`, `librsvg2-dev`, `build-essential`, `curl`, `file`, and `wget` via `apt-get` before invoking `tauri build`, and the build succeeds producing both an `.AppImage` and an `.AppImage.tar.gz` next to it.

#### Scenario: Windows build emits an unsigned MSI

- **WHEN** the `x86_64-pc-windows-msvc` matrix entry completes
- **THEN** the bundle directory contains a `.msi` installer and a `.msi.zip` updater archive with its `.sig`. The MSI is NOT Authenticode-signed in this iteration. The workflow MUST NOT fail due to the absence of a Windows signing certificate.

#### Scenario: One target failing does not block the others

- **WHEN** the Linux build fails (e.g. transient apt mirror outage) but mac and Windows builds succeed
- **THEN** the mac and Windows jobs complete, but the publish job does not run because not all four artifacts are present; the previous `latest.json` on R2 is untouched.

#### Scenario: Build failure leaves previous release intact

- **WHEN** any step (build, sign, notarize, upload) fails on any target
- **THEN** R2 still serves the previous `latest.json` and the team's installed apps continue to receive that previous version on their next update check.

### Requirement: Manifest is signed and contains per-architecture entries

The `latest.json` published to R2 MUST follow the Tauri v2 updater manifest schema with one `platforms` entry per supported target. The set of supported targets is: `darwin-aarch64`, `darwin-x86_64`, `linux-x86_64`, and `windows-x86_64`. Each entry MUST contain a `signature` field generated by the Ed25519 updater key over the corresponding updater archive (`.app.tar.gz` for macOS, `.AppImage.tar.gz` for Linux, `.msi.zip` for Windows), the `url` of that archive in R2, and the top-level `version` field MUST match the tag without the leading `v`.

The manifest generator (`scripts/build-manifest.mjs`) MUST accept per-platform tarball and signature env vars and SHALL omit any platform whose env vars are not provided, while still emitting a structurally valid manifest. The CI publish step MUST always provide all four platforms; the local script MAY provide a subset (whatever the host can build) and MUST log a warning when emitting a partial manifest.

#### Scenario: Manifest validates against installed app pubkey on every platform

- **WHEN** the installed app on macOS, Linux, or Windows (each embedding the same public Ed25519 key) fetches `latest.json` and the target archive for its platform key
- **THEN** the signature in the manifest validates against the archive bytes using the embedded public key, and the update proceeds.

#### Scenario: Tampered archive is rejected

- **WHEN** the archive at the URL has been modified after publication and the signature in `latest.json` no longer matches
- **THEN** the updater plugin refuses to apply the update and surfaces an error in the app log.

#### Scenario: Partial local manifest omits missing platforms

- **WHEN** a Linux developer runs the local script with `--target x86_64-unknown-linux-gnu` only (no mac/Windows artifacts available)
- **THEN** `scripts/build-manifest.mjs` writes a `latest.json` that contains only `linux-x86_64` under `platforms`, prints a warning that mac and Windows are missing, and the script refuses to upload that partial manifest unless the developer passes `--allow-partial-manifest`.

### Requirement: Manual setup is documented end-to-end

The repository SHALL include `docs/release-setup.md` that documents every manual step a human must perform once to bootstrap the pipeline, in the order they must be performed. The doc MUST cover: Apple Developer ID certificate creation and export to base64; app-specific password creation; Cloudflare account, R2 bucket creation, public-bucket toggling, and API token scoping; Ed25519 updater keypair generation via `pnpm tauri signer generate` and where to back up the private key; the exact list of GitHub repository secrets and their values; the icon variant generation procedure (placement under `src-tauri/icons-beta/`); the Linux runner system-package list and the rationale for pinning the Ubuntu version; and an explicit "Windows MSI is unsigned in this iteration" note describing the SmartScreen prompt users will see on first install. The doc MUST also include a rollback runbook describing how to publish a hand-crafted `latest.json` pointing at a previous good version when a release breaks the team.

#### Scenario: A teammate following the doc reaches a working release

- **WHEN** a teammate with no prior context follows `docs/release-setup.md` from top to bottom
- **THEN** they end up with a configured GitHub Actions workflow that successfully produces signed, notarized macOS releases, an unsigned Linux AppImage, and an unsigned Windows MSI on the next merge to `master`.

#### Scenario: Rollback runbook restores a known good version

- **WHEN** a release breaks the app and the rollback runbook in `docs/release-setup.md` is followed
- **THEN** R2's `latest.json` is rewritten to point at a previous version, all running team apps detect the "downgrade" on their next 4-hour check, and on the next quit they return to that version.

## ADDED Requirements

### Requirement: Local release script supports per-host target builds

`scripts/release-local.sh` SHALL detect the host operating system via `uname -s` and build only the targets that are natively supported on that host (no cross-compilation in the first iteration). The supported host-to-target mapping is:

- macOS host (`Darwin`): may build `aarch64-apple-darwin` and/or `x86_64-apple-darwin`.
- Linux host (`Linux`): may build `x86_64-unknown-linux-gnu`.
- Windows host (`MINGW*`/`MSYS*`/`CYGWIN*` under Git Bash, or PowerShell via an equivalent `release-local.ps1`): may build `x86_64-pc-windows-msvc`.

The script MUST:

- Skip Apple-only preflight checks (keychain identity, `codesign`, `xcrun`, Apple env vars) when the host is not macOS.
- Install or check the rustup target for the chosen platform and abort with a clear error if it cannot be installed.
- Run `tauri build` with the appropriate `--target` flag and collect the platform's native artifacts into `staging/` under the canonical name scheme `Argus_${VERSION}_${ARCH}.<ext>` where `<ext>` is `.dmg`/`.app.tar.gz`/`.app.tar.gz.sig` on mac, `.AppImage`/`.AppImage.tar.gz`/`.AppImage.tar.gz.sig` on Linux, and `.msi`/`.msi.zip`/`.msi.zip.sig` on Windows.
- Accept a new `--allow-partial-manifest` flag that lets the local script generate and optionally upload a `latest.json` containing only the platforms it built; without that flag, partial manifests are written to `staging/latest.partial.json` for inspection but never uploaded.

#### Scenario: macOS host builds both mac targets by default

- **WHEN** a developer on macOS runs `./scripts/release-local.sh` with no flags
- **THEN** the script builds `aarch64-apple-darwin` and `x86_64-apple-darwin`, performs the existing codesign + notarize flow, and (because Linux + Windows artifacts are missing) writes the manifest to `staging/latest.partial.json` and refuses to upload it.

#### Scenario: Linux host produces an AppImage

- **WHEN** a developer on Ubuntu 22.04 runs `./scripts/release-local.sh --target x86_64-unknown-linux-gnu --no-push --no-upload`
- **THEN** the script verifies the system `apt` packages are present (and prints the missing ones if not), builds the `.AppImage`, renames it to `Argus_${VERSION}_x64.AppImage`, and stages the corresponding updater tarball and signature alongside it.

#### Scenario: Windows host produces an MSI

- **WHEN** a developer on Windows runs the PowerShell equivalent of the local script with `-Target x86_64-pc-windows-msvc -NoPush -NoUpload`
- **THEN** the script builds the `.msi`, renames it to `Argus_${VERSION}_x64.msi`, stages the `.msi.zip` updater archive and `.sig`, and does NOT attempt to invoke `codesign` or any macOS-only tool.

#### Scenario: macOS host cannot accidentally build for Linux/Windows

- **WHEN** a developer on macOS passes `--target x86_64-unknown-linux-gnu`
- **THEN** the script aborts with a clear error message stating that cross-compilation to Linux is not supported by this script and points them at the GitHub Actions workflow as the cross-platform path.

### Requirement: A fixed-URL `download.json` exposes the latest installer URLs

The release pipeline SHALL publish a JSON document named exactly `download.json` to the root of the public R2 bucket on every successful release, served from the stable URL `${PUBLIC_URL_BASE}/download.json`. This file is the canonical public entry point for landing pages, README badges, and "Download Argus" buttons that need a direct link to the latest installer without knowing the current version number.

The document MUST be valid JSON conforming to this schema:

```json
{
  "version": "<X.Y.Z, no leading v>",
  "pub_date": "<ISO 8601 UTC timestamp>",
  "installers": {
    "darwin-aarch64": { "url": "<absolute URL to .dmg>", "filename": "<basename>", "size": <bytes> },
    "darwin-x86_64":  { "url": "...", "filename": "...", "size": <bytes> },
    "linux-x86_64":   { "url": "<absolute URL to .AppImage>", "filename": "...", "size": <bytes> },
    "windows-x86_64": { "url": "<absolute URL to .msi>", "filename": "...", "size": <bytes> }
  }
}
```

The `installers` block MUST point at the **end-user installer** (`.dmg` on macOS, `.AppImage` on Linux, `.msi` on Windows) — never at the updater archive (`.app.tar.gz` / `.AppImage.tar.gz` / `.msi.zip`). The `size` field reports the installer's byte length so the consumer can show a "Download (45 MB)" label.

The CI publish step MUST emit `download.json` *only* after all four installer artifacts pass the existing "all bundles present" verification, and MUST upload it with `Cache-Control: no-cache, max-age=0` and `Content-Type: application/json` — identical caching to `latest.json` so the URL is always live. The local script MUST be able to emit `download.json` for whatever subset of platforms it built, but MUST NOT upload it to R2 unless `--allow-partial-manifest` is passed (the same flag that gates partial updater manifests).

The fixed URL `${PUBLIC_URL_BASE}/download.json` MUST remain stable across all future versions — its path and filename are part of the public API of the release pipeline.

#### Scenario: Landing page fetches latest installer URL without knowing the version

- **WHEN** a landing page at argus.example.com loads and fetches `https://<r2-public>/download.json`
- **THEN** the response contains the version number of the most recent release plus four installer URLs (dmg arm64, dmg x64, AppImage, msi), each resolvable and downloading the actual installer bytes for that version.

#### Scenario: download.json points at installers, not updater archives

- **WHEN** the manifest generator emits `download.json` for release `v1.2.3`
- **THEN** `installers["darwin-aarch64"].filename` ends with `.dmg`, `installers["linux-x86_64"].filename` ends with `.AppImage`, `installers["windows-x86_64"].filename` ends with `.msi`, and NONE of the four URLs point at `.app.tar.gz`, `.AppImage.tar.gz`, `.msi.zip`, or any `.sig` file.

#### Scenario: download.json is served fresh, not cached

- **WHEN** two consecutive releases happen and a client fetches `download.json` after each
- **THEN** both fetches return the correct latest-version JSON (no stale CDN cache), because the upload step pinned `Cache-Control: no-cache, max-age=0`.

#### Scenario: CI refuses to publish an incomplete download.json

- **WHEN** any of the four installer files is missing from the staging directory at publish time
- **THEN** the CI workflow fails before uploading `download.json`, and the previous `download.json` on R2 remains the live source of truth.

#### Scenario: Local partial build writes download.partial.json instead of overwriting the live one

- **WHEN** a Linux developer runs the local script on a Linux host and builds only the Linux AppImage
- **THEN** the manifest generator writes the partial output to `staging/download.partial.json` (NOT `download.json`), the script refuses to upload it without `--allow-partial-manifest`, and the R2-hosted `download.json` is untouched.

### Requirement: Linux and Windows artifacts use canonical filenames

All published Linux and Windows artifacts SHALL use the same `Argus_${VERSION}_${ARCH}.<ext>` naming convention as the existing macOS artifacts so that the manifest's `url` fields are predictable and so that public R2 caching keys are stable across releases. The accepted extensions are:

- macOS: `.dmg`, `.app.tar.gz`, `.app.tar.gz.sig`
- Linux: `.AppImage`, `.AppImage.tar.gz`, `.AppImage.tar.gz.sig`
- Windows: `.msi`, `.msi.zip`, `.msi.zip.sig`

The CI workflow's collect step and the local script's stage step MUST both produce these exact names regardless of the upstream Tauri default names.

#### Scenario: CI artifact names match the manifest URLs

- **WHEN** CI publishes release `v1.2.3`
- **THEN** R2 contains, among others, `Argus_1.2.3_x64.AppImage` and `Argus_1.2.3_x64.msi`, and `latest.json`'s `platforms["linux-x86_64"].url` and `platforms["windows-x86_64"].url` point to those exact filenames under `PUBLIC_URL_BASE`.

#### Scenario: A version-by-version diff of R2 keys is human-readable

- **WHEN** an operator lists the R2 bucket
- **THEN** every artifact key follows the same `Argus_<version>_<arch>.<ext>` shape, making it trivial to grep by version or by platform.
